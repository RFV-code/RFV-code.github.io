Abstraction, the Anti-Pattern of Computation

Today, after much thought again, I realized that abstraction is in fact the anti-pattern of computation. In computer programming there are tones of ideas, frameworks, platforms, APIs, languages, tools, methodologies, etc, etc and yet at the very heart of it all lies computation and more specifically the computer. A computer is a real engineered physical machine that performs exactly as it was designed to do, so surely the field of computer engineering should be more, well, structured and organized?

The issue lies with abstraction. Abstraction is a good tool, but if you don't understand how to use that tool then you won't know what your tool is doing and what your actually busy with. The more abstract you become the less you think about whats actually happening.

Abstraction should only be used when you know how that abstraction relates the world without that abstraction, in other words you need to know what yous abstraction is doing.

Computation in itself is something which is very much a real thing as it always requires a real machine and a program runs in real time..

Abstraction is like a great magic trick. If one wishes to fully engage or understand the trick, one needs to know how it is performed, and if you don't really want to spoil the magic, then at the very least: you have to remember that it's just a magic trick. It's not real.

Real world targets, deadlines, and then neutrally people who have flaws, egos, tastes, etc, etc is the real reason that the world of software is in the state that it is today. Although I'm not expecting it to happen, I do think that future generations will benefit if everyone takes a little bit of time to find out what everyone else is busy d. But since the world of software development is not driven by making things work better (better under the hood, and more efficient), just as the rest of the world is not driven by making things more efficient and more sustainable (engineering, energy, logistics), but by many factors such as profits, politics, etc. Just so the world of software development will linger on in a state of mess. The thing is though, that software in itself (not the running of software, but the data representation of software) is such a pure and concrete thing, that I believe it time for Universities to really take a pause and re-evaluate what we are busy with. CPU's are not simply going to keep getting better.

The common denominator of the computer of today (PC, Server, Mobile, Raspberry Pi) is the Von Neumann Architecture which in turn uses the concept of the Turing machine.

The current world of program abstractions.

To understand your abstractions (in the imperative programming world) you need to understand the evolution of software engineering abstraction, and in parallel the evolution of programming languages in order of:

  1. Turing Machine (3 basic statements: conditional statement [if-then], jump [move execution to another position], change [do a combinatory logic execution])
  2. Variables and data structures (such as int, string, or any block of sequential data)
  3. The procedure (which requires a stack)
  4. Removal of the Goto statement (since procedures can structure code into blocks of tree structures) *LANGUAGE FEATURE REDUCING COMPUTATION ABILITY - no longer Turing complete
  5. Run-time: dynamic memory management and allocation, pointers (run time allocated memory address)
  6. Objects: grouping procedures and variables together and enabling easy dynamic allocation of new such objects. Procedures have limited access to its own variables and are called methods. *LANGUAGE FEATURE REDUCING COMPUTATION ABILITY (syntactic sugar: makes life easy for doing common things, when used)
  7. Exceptions, Garbage Collection, Dynamic Typing, Run-time compilation, etc, etc *LANGUAGE FEATURE REDUCING COMPUTATION ABILITY (syntactic sugar: makes life easy for doing common things, when used) These are all good default features for a language to have but since they are not optional (not extensions, but replacements), and mandatory they actually reduce the overall possible computation possible.

I believe that we are capable of much more amazing computational application then we even realize. The software world seems to move in this monolithic way where we tend to crate thing that we already know, leading to the same old same old. I am not an expert in any language, and that doesn't matter for me to be able to write this article. Perhaps it is because I am not an expert and tied down to any specific way of thinking that I am able to state here, what I think many programmer already intuitively know by themselves.
